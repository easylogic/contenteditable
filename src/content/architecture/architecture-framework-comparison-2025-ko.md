## 개요
리치 텍스트 에디터를 구축할 때 기반이 되는 프레임워크를 선택하는 것은 매우 중요한 결정입니다. 2025년 현재, 시장은 서로 다른 아키텍처 요구 사항에 적합한 세 가지 주요 경쟁자로 안정화되었습니다.

## 비교 테이블

| 기능 | Lexical | ProseMirror | Slate |
| :--- | :--- | :--- | :--- |
| **철학** | 성능 우선, 확장성 중심 | 스키마 기반, 함수형 | 빌더 키트, 플러그인 위주 |
| **모델** | 가변 노드 트리 (클론된 상태 사용) | 불변 JSON 형태의 트리 | JSON 기반 중첩 객체 |
| **조정 (Reconciler)** | Diff 알고리즘 (React 방식) | 직접적인 DOM 동기화 | React 네이티브 렌더링 |
| **협업 (OT/CRDT)** | 외부 라이브러리 (yjs 등) 필요 | 네이티브 지원 (매우 견고함) | 외부 구현 필요 (커스텀 필요) |
| **접근성** | 최고 수준 (Meta 주도) | 우수함 (성숙한 생태계) | 수동 ARIA 설정 필요 |
| **학습 곡선** | 보통 | 높음 (함수형 개념 생소) | 보통에서 높음 사이 |

## 상세 분석

### 1. Lexical (현대적인 표준)
Meta에서 개발한 Lexical은 속도와 신뢰성에 최적화되어 있습니다. **React 우선**으로 설계되었지만 프레임워크에 구애받지 않는 구조를 가집니다.
- **장점**: 가장 작은 번들 사이즈(~22kb), 모바일에서의 뛰어난 성능, 강력한 접근성 기본 설정.
- **단점**: 2025년 초 기준 여전히 v1 미만이며, API 상에서 노드를 복제하고 업데이트하는 보일러플레이트 코드가 다소 존재함.
- **추천 대상**: 커스텀 소셜 미디어 에디터, 성능이 중요한 SaaS 도구.

### 2. ProseMirror (검증된 엔진)
사용 준비가 된 에디터라기보다는 '에디터를 만들기 위한 툴킷'에 가깝습니다. 매우 구조적이며 **스키마**를 통해 모든 유효한 문서 상태를 정의합니다.
- **장점**: 복잡한 협업 워크플로우에서 가장 신뢰할 수 있음, 문서 변형에 대한 절대적인 제어권 제공.
- **단점**: 학습 곡선이 매우 높음 (Commands, Plugins, Transforms 개념), React의 비동기 렌더링과 네이티브하게 잘 맞지 않음(Tiptap과 같은 래퍼 사용 권장).
- **추천 대상**: 구글 문서 스타일의 협업 도구, 법률 문서, 학술 작성 도구.

### 3. Slate (커스터마이징의 정수)
모든 기능을 플러그인으로 취급하여 극강의 유연성을 제공합니다.
- **장점**: 수학 공식, 복잡한 테이블 등 특수 노드를 구축하기 매우 쉬움, React 개발자에게 친숙한 구조.
- **단점**: 대용량 문서에서 성능 저하가 발생할 수 있음, 대규모 팀에서 플러그인 관리가 복잡해질 수 있음.
- **추천 대상**: 독특한 UI 요구 사항이 있는 틈새 시장향 에디터, 프로토타이핑.

## 빠른 선택 가이드
- **"빠르고 바로 작동하는 React 에디터가 필요해요."** -> [Tiptap](https://tiptap.dev/) (ProseMirror 기반) 또는 Lexical.
- **"차세대 협업 문서 도구를 만들고 싶어요."** -> ProseMirror.
- **"기존에 없던 독특한 시각적 블록을 가진 에디터가 필요해요."** -> Slate.
